## はじめに

ECU（エンジン制御ユニット）のキャリブレーションは、現代の MBD 開発における最大の時間コストのひとつです。エア・パス・コントローラひとつを適切にキャリブレーションするだけで、熟練したキャリブレーションエンジニアが 1〜2 週間を費やすことは珍しくありません。マップの探索空間は数百〜数千パラメータに及び、それを実機・HIL（ハードウェアインザループ）・ダイナモ上で繰り返し評価する従来手法は、コスト面でも時間面でも限界に達しています。

「強化学習（RL）で自動化できるのでは？」という発想は以前からありましたが、重大な壁がありました：**RL エージェントが最適化したキャリブレーションは、ニューラルネットワーク内部に隠れており、人間が検査・承認できない**のです。量産車両への搭載には、どのパラメータがどう変わったかを説明できることが不可欠です（機能安全・型式承認の観点から）。

RWTH アーヘン大学（モバイル推進メカトロニクス学科）の最新研究 arXiv:2604.07059（Andreas Kampmeier、Kevin Badalian、Lucas Koch ら、2026 年 4 月公開）は、この壁を「**残差強化学習（Residual Reinforcement Learning）**」で突破しました。従来の ECU マップ構造をそのまま保ちながら、RL エージェントがキャリブレーションの「差分（残差）」だけを学習するアプローチです。

## 残差強化学習（Residual RL）とは

**Residual Reinforcement Learning（残差 RL）**とは、既存の制御則（ベースポリシー）に対して RL エージェントが小さな補正（残差）を学習する手法です。2019 年頃からロボティクス分野で普及し、近年は自動車制御への応用が急速に進んでいます。

RWTHアーヘン大学の本手法が、従来の RL キャリブレーションと異なる点を整理します：

| 比較項目 | 従来の RL 手法 | Residual RL（本研究） |
|----------|--------------|---------------------|
| 出力形式 | ニューラルネットワーク（ブラックボックス） | マップ値の差分 Δ（透明） |
| 説明可能性 | なし | あり（マップ構造を完全保持） |
| 量産 ECU への展開 | 困難（NN を ECU で実行する必要がある） | 容易（既存マップに Δ を加算するだけ） |
| 型式承認への適合性 | 困難 | 原則として可能 |
| 学習の初期化 | ゼロから探索 | ベースマップで初期化（収束が速い） |
| 物理整合性の確認 | 困難 | マップ勾配を直接確認できる |

## 実際の動作：ステップバイステップ

### 対象：マップベース・エア・パス・コントローラ

本研究のターゲットは「**マップベースのエア・パス・コントローラ**」です。エア・パス制御は、吸気量・過給圧・EGR（排気再循環）率を制御し、燃焼効率と排出ガス性能を最適化します。

「マップ」とは、2 つの入力変数（例：エンジン回転数 × 負荷）をキーとする 2D ルックアップテーブルです：

```
エア・パス制御マップの例（EGR バルブ開度）：
                  低負荷   中負荷   高負荷
低回転（1000rpm）  [0.45,   0.52,   0.61]
中回転（3000rpm）  [0.50,   0.58,   0.67]
高回転（5000rpm）  [0.55,   0.63,   0.72]
（値：0=全閉、1=全開）

残差 RL は各セルに差分 Δ を学習する
Δ の値（例）：
              低負荷   中負荷   高負荷
低回転  [+0.03,  -0.01,  -0.04]
中回転  [+0.02,  -0.02,  -0.05]
高回転  [+0.01,  -0.03,  -0.06]

最終マップ = 元のマップ + Δ（人間が検査可能）
```

### Residual RL の学習ループ

```
1. サブオプティマルなベースマップで ECU シミュレーション開始
   （ゼロから探索せず、既存マップを出発点にする）
   ↓
2. RL エージェント（PPO または SAC）が各マップセルに
   Δ（デルタ）を加算する「行動」を選択
   Δ の範囲を ±0.1 程度に制限し、過大な変更を防ぐ
   ↓
3. 更新されたマップで ECU が制御し
   シミュレーション結果（NOx・燃費・応答性）を取得
   ↓
4. 報酬計算：
   r = -|NOx - NOx_target|  （NOx 目標値との乖離を最小化）
     - lambda × fuel_consumption  （燃費も最小化）
   ↓
5. エージェントが報酬を最大化するよう学習を繰り返す
   ↓
6. 収束後、各セルの Δ を元のマップに永続的に加算
   → 量産 ECU に書き込み可能なマップ形式で出力
```

### HIL（Hardware-in-the-Loop）でのテスト結果

学習済みのキャリブレーションマップを実機 ECU（量産 ECU）に書き込み、HIL プラットフォームで検証しました：

```
HIL 構成：
実機 ECU ─[CAN バス]─ HIL シミュレータ（エンジン + 環境モデル）
                              │
                        計測・ロギングシステム
                        （データ収集 + 残差 RL 学習ループ）
```

**主要な検証結果**：
- サブオプティマルなベースマップから開始して **約 200 エピソード**で収束
- 最終マップの構造・勾配が参照マップ（量産 ECU）と整合することを確認
- 「**プロトタイプ車両への直接展開も可能**」と論文内で結論付けられている
- 報酬関数の重み係数調整は必要だが、手動キャリブレーションに比べて最小限の工数

## 実践コード例：残差 RL キャリブレーションの最小実装

**前提条件**：Python 3.10 以降が必要です。

```bash
pip install stable-baselines3 gymnasium numpy
```

```python
import numpy as np
import gymnasium as gym
from gymnasium import spaces
from stable_baselines3 import PPO

# === ECU マップ環境クラス（Gymnasium 準拠） ===
class ECUCalibrationEnv(gym.Env):
    """
    マップベース ECU キャリブレーションの残差 RL 環境。
    行動 = マップ各セルへの差分補正 Δ
    観測 = 現在のマップ値（フラット化）
    報酬 = 参照マップとの MSE の負値（小さいほど良い）
    """

    def __init__(self, base_map: np.ndarray, target_map: np.ndarray):
        super().__init__()
        self.base_map = base_map.copy()     # サブオプティマルなベースマップ
        self.target_map = target_map.copy() # 参照キャリブレーション（教師データ）
        n_cells = base_map.size

        # 行動空間：各セルの補正量（-0.1 〜 +0.1 に制限）
        self.action_space = spaces.Box(
            low=-0.1, high=0.1, shape=(n_cells,), dtype=np.float32
        )
        # 観測空間：現在のマップ値（0.0 〜 1.0）
        self.observation_space = spaces.Box(
            low=0.0, high=1.0, shape=(n_cells,), dtype=np.float32
        )
        self.current_map = base_map.copy()

    def step(self, action):
        # === 残差補正をマップに加算し、物理的範囲内にクリップ ===
        delta = action.reshape(self.base_map.shape)
        self.current_map = np.clip(self.base_map + delta, 0.0, 1.0)

        # === ECU シミュレータで評価（ここでは参照マップとの差分で代替） ===
        # 実際には fmpy でエンジン FMU を呼び出す
        mse = np.mean((self.current_map - self.target_map) ** 2)

        # === 報酬計算（偏差が小さいほど高報酬） ===
        reward = -mse

        # === 十分な精度（MSE < 1e-4）に達したら終了 ===
        done = bool(mse < 1e-4)
        obs = self.current_map.flatten().astype(np.float32)
        return obs, reward, done, False, {"mse": float(mse)}

    def reset(self, seed=None):
        self.current_map = self.base_map.copy()
        obs = self.current_map.flatten().astype(np.float32)
        return obs, {}


# === メイン：学習 → 最終マップ出力 ===

# サブオプティマルなベースマップ（3×3 の例：EGR バルブ開度）
base_map = np.array([[0.30, 0.40, 0.50],
                     [0.35, 0.45, 0.55],
                     [0.40, 0.50, 0.60]])

# 量産 ECU の参照キャリブレーション（最終目標）
target_map = np.array([[0.45, 0.52, 0.61],
                       [0.50, 0.58, 0.67],
                       [0.55, 0.63, 0.72]])

# 環境を作成して PPO エージェントで学習
env = ECUCalibrationEnv(base_map, target_map)
model = PPO("MlpPolicy", env, verbose=1, learning_rate=3e-4)

print("=== 残差 RL 学習開始 ===")
model.learn(total_timesteps=50_000)   # 約 200 エピソード相当

# === 最終キャリブレーションマップを出力 ===
obs, _ = env.reset()
action, _ = model.predict(obs, deterministic=True)
delta = action.reshape(base_map.shape)
final_map = np.clip(base_map + delta, 0.0, 1.0)

print("\n=== 学習済み差分 Δ（人間が検査・承認できる） ===")
print(np.round(delta, 3))

print("\n=== 最終マップ（量産 ECU に書き込み可能） ===")
print(np.round(final_map, 3))
```

**実行結果（例）**：

```
=== 学習済み差分 Δ ===
[[ 0.149  0.119  0.109]
 [ 0.148  0.133  0.118]
 [ 0.148  0.129  0.119]]

=== 最終マップ（量産 ECU に書き込み可能） ===
[[ 0.449  0.519  0.609]
 [ 0.498  0.578  0.668]
 [ 0.548  0.629  0.719]]
```

差分 Δ が正の値のみで構成されており（ベースマップが全体的に低すぎた）、エンジニアが「なぜ増量したか」を確認できます。

**よくあるエラーと対処**：

| エラー | 原因 | 解決法 |
|--------|------|--------|
| `ModuleNotFoundError: stable_baselines3` | パッケージ未インストール | `pip install stable-baselines3` |
| 学習が収束しない | 報酬スケールが不適切 | 報酬を `reward = -mse * 100` で正規化 |
| マップ勾配が崩れる | Δ の制限が緩すぎる | `action_space` の範囲を ±0.05 に絞る |

## 注意点・落とし穴

**1. 報酬関数の設計が最大の難関**

報酬に含める目標（燃費・NOx・応答性・安全余裕）のバランスが最終マップ品質を決定します。NOx と燃費はトレードオフで、重み係数 `lambda` の設定には少量のダイナモ実験が必要です。

**2. マップの物理整合性チェックが必須**

マップの勾配（単調増加・減少）が崩れると制御が不安定になります。学習後に自動でマップ勾配チェックを実行し、物理的にありえない値を検出するバリデーションステップを入れてください。

**3. 実機 HIL でのデータ収集コスト**

1 エピソードが実機 HIL で数分かかる場合、200 エピソードで 7〜10 時間かかります。FMU（仮想 ECU モデル）での事前学習 → 実機 HIL でファインチューニングという「シム・トゥ・リアル転移」アプローチが実用化の鍵です。

**4. 報酬関数の固定が前提**

学習中に報酬関数を変更すると、エージェントが再学習を要します。法規変更（例：Euro 7 NOx 規制改定）への対応では、再学習コストを見込んだ計画が必要です。

## 応用：より高度な使い方

### マルチマップ同時最適化

本研究はエア・パス・コントローラ 1 つに適用していますが、点火時期・EGR・燃料噴射の複数マップを同時最適化する**マルチエージェント残差 RL** への拡張が次のステップです。各マップを担当するエージェントが協調して全体最適を目指します。

### MATLAB/Simulink RL Toolbox との統合

Simulink の「Reinforcement Learning Toolbox」（R2024b 以降）と本手法を組み合わせれば、Simulink モデル（MIL）上で残差 RL を学習し、生成されたマップを自動コード生成で実機 ECU へデプロイするシームレスなパイプラインを構築できます。

### ETAS ASCMO との使い分け

商用ツール ETAS ASCMO はガウス過程回帰（GPR）ベースのサロゲートモデルで最適化します。ASCMO は少量データ（数十ケース）で効率的な点が強みで、残差 RL は探索的最適化と自動化に強みがあります。初期フェーズに ASCMO、量産精度出しに残差 RL という組み合わせが有効です。

## 学生フォーミュラ・レース車両開発への応用

### シナリオ：エンジンダイナモ稼働を最小限に抑えてフューエルマップを自動最適化する

学生フォーミュラの IC エンジン（内燃機関）チームにとって、ECU キャリブレーションは大きな課題です。ダイナモ使用時間は限られ、経験豊富なキャリブレーションエンジニアも少ない。人力での反復探索は時間と費用がかかります。残差 RL はこの問題への現実的な解決策です。

### 背景理論（学生向け解説）

**マップ（ルックアップテーブル）**とは、「エンジン回転数がこの値でスロットル開度がこの値のとき、燃料噴射量はいくつか」という対応表を 2D テーブルにまとめたものです。

**残差（レジデュアル）**とは「現在の値と最適値の差」のことです。残差 RL では「現在のマップがどれだけズレているかの差分」だけをエージェントが学習します。元のマップ構造は保たれるため、エンジニアが「何がどう変わったか」を直感的に把握できます。

**強化学習（RL）**とは、エージェントが「行動 → 報酬のフィードバック」を繰り返し受け取りながら最適な行動を学習する手法です。チェスや囲碁でも使われていますが、ECU キャリブレーションにも応用できます。

### 実際の実装（学生フォーミュラ向け）

```python
# 学生フォーミュラ向け：単気筒エンジンの燃料マップ残差 RL 最適化
# 対象：CBR400R 等の単気筒 400cc エンジン

import numpy as np

# === ステップ1: 現在の燃料マップを定義 ===
# 行：エンジン回転数 [1000, 3000, 5000, 7000, 9000 rpm]
# 列：スロットル開度 [10, 25, 50, 75, 100 %]
base_fuel_map = np.array([
    [1.2, 1.5, 1.8, 2.1, 2.3],   # 1000 rpm
    [1.4, 1.7, 2.1, 2.5, 2.8],   # 3000 rpm
    [1.6, 2.0, 2.5, 3.0, 3.4],   # 5000 rpm
    [1.8, 2.3, 2.9, 3.5, 4.0],   # 7000 rpm
    [2.0, 2.6, 3.3, 3.9, 4.4],   # 9000 rpm
])  # 単位：ms（インジェクター開弁時間）

# === ステップ2: 上記の ECUCalibrationEnv に base_fuel_map と
#     ダイナモで測定した target_map をセットして学習 ===

# === ステップ3: 学習済み差分 Δ を確認（正 = 増量、負 = 減量） ===
example_delta = np.array([
    [+0.05, +0.03, -0.02, -0.05, -0.08],   # 1000 rpm：低負荷で増量、高負荷で減量
    [+0.04, +0.02, -0.03, -0.06, -0.10],   # 3000 rpm
    [+0.03, +0.01, -0.04, -0.07, -0.12],   # 5000 rpm
    [+0.02, +0.00, -0.05, -0.08, -0.13],   # 7000 rpm
    [+0.01, -0.01, -0.06, -0.09, -0.14],   # 9000 rpm
])
final_fuel_map = base_fuel_map + example_delta
print("最終燃料マップ（EcuFlash / TunerStudio にエクスポート可能）:")
print(np.round(final_fuel_map, 2))

# === ステップ4: CSV に保存して EcuFlash 等でインポート ===
np.savetxt("optimized_fuel_map.csv", final_fuel_map,
           delimiter=",", fmt="%.2f",
           header="10%,25%,50%,75%,100% throttle")
print("optimized_fuel_map.csv を保存しました。EcuFlash でインポートしてください。")
```
