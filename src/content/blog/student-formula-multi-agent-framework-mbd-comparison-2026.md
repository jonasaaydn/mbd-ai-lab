## 学生フォーミュラにおける課題

多くのチームは「Simulinkでモデル作成 → MATLAB Scriptで解析 → Wordで報告書作成」という手順を手動で繰り返しています。この作業は1サイクルあたり平均**3〜4時間**を要し、大会直前期には週5回以上発生します。担当者が卒業すると手順が口頭伝承となり、引き継ぎコストも大きいです。また「誰が今どのモデルバージョンで作業しているか」の同期に毎回15〜30分かかります。マルチエージェントフレームワークを使えば、この繰り返し作業を「自然言語で指示するだけ」で自動化できます。

## マルチエージェントフレームワークを使った解決アプローチ

マルチエージェント（複数のAIエージェントが協調して作業を分担する仕組み）では、1つのAIが「作業を分解して指示するオーケストレーター（指揮者）」、複数のAIが「実際に作業する専門家エージェント」として動作します。

フレームワーク別の特徴と学生チームへの推奨度を下表にまとめます。

| フレームワーク | 設定の簡単さ | 条件分岐・ループ | 向いている用途 | 推奨度 |
|--------------|------------|---------------|--------------|-------|
| **CrewAI 0.80** | ★★★（最も簡単） | △（基本的な直列・並列のみ） | 定型レポート生成・解析サマリー | 入門向け ◎ |
| **LangGraph 0.2** | ★★☆ | ◎（柔軟なDAGで表現） | 複数ツール呼び出しが必要なワークフロー | 中級向け |
| **AG2（旧AutoGen）2.0** | ★★☆ | ○ | AIどうしの議論で精度を高めたい場合 | 中級向け |

学生チームには**CrewAI**が最初の一択です。設定がロール定義（「サスペンション解析担当」「レポート作成担当」など）だけで始められるためです。

## 実装：ステップバイステップ

**前提条件:** Python 3.11+, `pip install "crewai>=0.80" anthropic`, Claude API キー（`ANTHROPIC_API_KEY`）

```python
# === ステップ1: CrewAI のインストール確認 ===
# pip install "crewai>=0.80" anthropic
# 確認: python -c "import crewai; print(crewai.__version__)"

import os
from crewai import Agent, Task, Crew, Process

# APIキーを環境変数にセット（.envファイルや os.environ で管理する）
os.environ["ANTHROPIC_API_KEY"] = "sk-ant-..."  # 実際のキーに置き換える

# === ステップ2: 専門エージェントを定義 ===

# サスペンション解析担当エージェント
analyst_agent = Agent(
    role="サスペンション解析エンジニア",
    goal="MATLABシミュレーション結果を読み込み、問題点と推奨改善を特定する",
    backstory=(
        "学生フォーミュラチームのMBD担当メンバー。"
        "Simulinkモデルのパラメータ変更と結果解釈が専門。"
        "数値は必ず単位付きで説明する。"
    ),
    llm="claude-sonnet-4-6",  # 最新Claudeを使用
    verbose=True,
    allow_delegation=False,
)

# テクニカルレポーター担当エージェント
reporter_agent = Agent(
    role="テクニカルレポーター",
    goal="解析結果を学生フォーミュラ大会の技術審査官向けに簡潔に要約する",
    backstory=(
        "テクニカルライティング専門。"
        "数字と箇条書きで説明し、300字以内にまとめる。"
    ),
    llm="claude-sonnet-4-6",
    verbose=True,
    allow_delegation=False,
)

# === ステップ3: タスクを定義 ===

analysis_task = Task(
    description=(
        "以下のサスペンションシミュレーション結果を解析してください:\n\n"
        "変更内容:\n"
        "  - フロントロールセンター高さ: 52mm → 48mm\n"
        "  - フロントバンプステア: -0.8 deg/100mm → -0.6 deg/100mm\n"
        "  - スタティックキャンバー: -2.0° → -2.5°\n\n"
        "この変更がハンドリング特性（ロール応答・直進安定性・コーナリング限界）に"
        "与える影響を3点に絞って説明してください。"
        "各点は「変更→影響→理由」の形式で記述すること。"
    ),
    agent=analyst_agent,
    expected_output="3点のハンドリング影響と推奨事項（各点50〜80字）",
)

report_task = Task(
    description=(
        "前タスクの解析結果をもとに、大会技術審査官向けの"
        "サスペンション設計変更レポートを作成してください。\n\n"
        "フォーマット（Markdown）:\n"
        "## サスペンション設計変更レポート — {日付}\n"
        "**変更概要**: （2文以内）\n"
        "**技術根拠**:\n"
        "- 根拠1\n"
        "- 根拠2\n"
        "- 根拠3\n"
        "**実走テスト計画**: （2行以内）\n\n"
        "合計300字以内で作成すること。"
    ),
    agent=reporter_agent,
    context=[analysis_task],   # 前タスクの結果を引き継ぐ
    expected_output="300字以内の技術審査レポート（Markdown形式）",
)

# === ステップ4: Crew を実行 ===
crew = Crew(
    agents=[analyst_agent, reporter_agent],
    tasks=[analysis_task, report_task],
    process=Process.sequential,   # タスクを順番に実行（直列処理）
    verbose=True,
)

result = crew.kickoff()   # 実行開始（約30〜90秒）
print("\n=== 生成されたレポート ===")
print(result.raw)
```

**実行結果（例）:**
```
=== 生成されたレポート ===
## サスペンション設計変更レポート — 2026-06-22

**変更概要**: フロントロールセンターを4mm下げ、バンプステア改善とキャンバー増加を
同時に実施した設計変更。コーナリング応答の均一化を主目的とする。

**技術根拠**:
- ロールセンター低下によりロールモーメントアームが増加し、コーナリング時の
  初期応答遅れが改善（低速コーナーでの切り返し性向上を期待）
- バンプステア減少（0.2 deg/100mm）により縁石通過時の意図しない操舵が低減、
  直進安定性が向上
- ネガティブキャンバー増加（0.5°）によりコーナリング時のタイヤ接地圧均一化を図る

**実走テスト計画**: スラロームでバンプステア改善を確認（ドライバー主観評価5段階→4以上）。
定常円旋回でロールゲイン変化を計測（目標: 現状比±10%以内）。
```

## よくあるエラーと対処

| エラー | 原因 | 対処 |
|--------|------|------|
| `APIError: 429 Too Many Requests` | API呼び出し過多 | `Crew(max_rpm=10)`で速度制限 |
| エージェントが無限ループする | `expected_output`が曖昧 | 文字数・形式を具体的に指定する |
| `context`の引き継ぎが空になる | タスク定義順序のミス | `context=[前タスク変数]`を明示 |
| LangGraph版で`StateGraph`エラー | `typing_extensions`バージョン問題 | `pip install langgraph>=0.2`で再インストール |
| 日本語出力が文字化け | ターミナルのエンコード | `PYTHONUTF8=1`を環境変数に追加 |

## 学生フォーミュラ・レース車両開発への応用

### 具体的なシナリオ

全日本学生フォーミュラ大会では、設計変更ごとに「変更根拠→シミュレーション→技術資料作成」のサイクルが発生します。特に大会1ヶ月前の集中変更期は、担当者が毎晩この手順を繰り返し、翌朝のチームミーティングで結果を報告します。CrewAIを使えば「解析担当AI」「文書作成AI」「チェックAI」を分業させ、人間は最終確認のみに集中できます。

### 背景理論

マルチエージェントが有効な理由は「分業の原則」にあります。1つのLLM（大規模言語モデル）に「解析もレポートも全部やれ」と指示するより、「解析専門AI」と「文書専門AI」に分業させる方が出力品質が高い傾向があります。CrewAIの`backstory`フィールドがこの役割設定を担います。LangGraphはこれを有向グラフ（DAG、Directed Acyclic Graph）で表現し、条件によって次に実行するエージェントを切り替えられます。

### 実際に動くコード（3エージェント構成への拡張）

```python
# 解析 → 品質チェック → レポート の3段構成
from crewai import Agent, Task, Crew, Process

# QAチェック担当エージェントを追加
qa_agent = Agent(
    role="品質保証レビュアー",
    goal="解析結果の数値に矛盾がないかチェックし、不整合を指摘する",
    backstory="MBDの品質保証経験者。数値の単位・符号・物理的妥当性を厳密に確認。",
    llm="claude-sonnet-4-6",
    verbose=False,
    allow_delegation=False,
)

qa_task = Task(
    description=(
        "analysis_taskの解析結果を受け取り、以下を確認してください:\n"
        "1. 物理単位が正しいか（deg, mm, MPa など）\n"
        "2. ロールセンター変化とロールモーメントの関係が論理的に正しいか\n"
        "3. バンプステアの符号（正/負）が方向と一致しているか\n"
        "問題があれば具体的に指摘し、問題なければ「OK」とだけ返答する。"
    ),
    agent=qa_agent,
    context=[analysis_task],
    expected_output="「OK」または具体的な問題点の指摘（100字以内）",
)

# 3エージェント構成のCrewを実行
crew_3 = Crew(
    agents=[analyst_agent, qa_agent, reporter_agent],
    tasks=[analysis_task, qa_task, report_task],
    process=Process.sequential,
    verbose=True,
)
result_3 = crew_3.kickoff()
print(result_3.raw)
```

### Before / After（3エージェント構成）

| 指標 | 1エージェント | 3エージェント構成 |
|------|-------------|----------------|
| レポート生成時間 | 約40秒 | 約90秒 |
| 単位・符号の誤り検出率 | 低（自己チェックのみ） | 高（QAエージェントが独立レビュー） |
| 人間レビュー所要時間 | 30分 | 15分（AIが事前チェック済み） |

### 学生チームが今すぐ試せる最初のステップ

`pip install "crewai>=0.80" anthropic`を実行し、ステップ2〜4のコードをそのままコピーして実行してください。まずAPIキーの設定が通るかだけ確認すれば十分です。実際のシミュレーション結果データへの差し替えは動作確認後で問題ありません。

## 今週の学生チームへの宿題

最後に実施したシミュレーション変更の「変更概要」を3行で書き出し、それをステップ3の`analysis_task`の`description`に貼り付けて実行してください。AIが生成したレポートを先輩に見せてフィードバックをもらうだけで、フレームワークの有用性が実感できます。
