## はじめに

境界条件の設定ミスで24時間のFEA計算がやり直しになった経験はないだろうか。重力方向を間違えたまま走らせた構造解析、符号が逆のアクチュエータモデル、「安全側」と思っていた荷重条件が実は危険側だったケース――これらはすべて**多段階の工学的推論の失敗**に起因する。

2025年12月にOpenAIがリリースした**o3**は、従来のGPT-4oとは根本的に異なる「推論モデル」だ。内部で数百ステップの思考プロセス（内部CoT）を実行してから回答を生成するため、多段階の計算・検証・矛盾検出が大幅に向上した。MBDエンジニアが毎日直面する「ちょっと複雑な工学問題」の壁を一気に取り除く可能性がある。今すぐAPIで使える実践ガイドを解説する。

## OpenAI o3とは

OpenAIが2025年12月に発表し、API一般公開を2025年4月から行っている推論特化型モデル。内部的に「思考時間」を確保して複数の推論ステップを自動実行し、最終的な回答を生成する。通常のGPT-4oが「考えながら答える」とすれば、o3は「十分に考えてから答える」モデルだ。

**GPT-4oとo3の主な違い：**

| 指標 | GPT-4o | o3-mini | o3（フル） |
|------|--------|---------|-----------|
| 内部CoT | なし | あり | あり |
| ARC-AGI スコア | 50.3% | — | 96.7% |
| AIME 2024 スコア | 9.3% | 87.7% | — |
| 平均応答時間 | 3秒 | 10〜30秒 | 30〜90秒 |
| コスト（入力1Mトークン） | $5 | $1.1 | $15 |

APIには `reasoning_effort` パラメータ（low/medium/high）があり、速度とコストのトレードオフを用途に合わせて調整できる。

## 実際の動作：ステップバイステップ

**前提条件**

Python 3.10以降が必要。以下のコマンドでインストールできる。

```bash
pip install openai
# 環境変数にAPIキーをセット（コードに直書きしてはいけない）
export OPENAI_API_KEY="sk-..."
```

**ユースケース1：FEA境界条件の検証**

FEAモデルのセットアップをo3に説明し、境界条件の妥当性を確認させる実例を示す。

```python
# === ステップ1: OpenAIクライアントを準備する ===
# APIキーは環境変数 OPENAI_API_KEY から自動で読まれる
from openai import OpenAI
client = OpenAI()

# === ステップ2: FEA設定の検証プロンプトを作成する ===
fea_setup = """
以下のFEA設定を検証してください：
- 解析種別：片持ち梁の静的構造解析
- 材料：CFRP（E1=135GPa, E2=10GPa, ν12=0.3）
- 境界条件：左端を完全固定（6自由度拘束）
- 荷重：右端に集中荷重 F=500N（Z方向 上向き正）
- 要素タイプ：Shell要素（厚さ3mm）
- 梁寸法：長さ500mm × 幅50mm
問題点があれば全て指摘し、修正案を示してください。
"""

# === ステップ3: o3-miniで推論させる ===
# reasoning_effort="high" で詳細な推論を要求する
response = client.chat.completions.create(
    model="o3-mini",
    reasoning_effort="high",   # low/medium/high を選択
    messages=[
        {"role": "user", "content": fea_setup}
    ]
)

# === ステップ4: 返答と思考トークン数を表示する ===
print(response.choices[0].message.content)
usage = response.usage
print(f"思考トークン数: {usage.completion_tokens_details.reasoning_tokens}")
```

**実行すると、o3-miniはこのような応答を返す：**

```
境界条件に以下の問題点を検出しました：

1. 荷重方向の整合性が不明：Shell要素でZ方向「上向き正」とありますが、
   重力方向との整合性を確認してください。水平配置の梁なら重力はZ負方向です。
   荷重と重力が同方向なら危険側、逆方向なら安全側の設定となります。

2. CFRP異方性に積層方向未指定：E1とE2が定義されていますが、
   繊維方向角度θがセットアップに含まれていません。θ=0°（梁長手方向）
   であれば高剛性ですが、θ=90°では著しく低剛性になります…
```

GPT-4oでは「問題なさそうです」と回答した設定でも、o3は矛盾点を2〜5個列挙する。

## 実践コード例

**ISO 26262 ASILアセスメント支援の実装：**

```python
# === ブレーキシステムのASIL分類をo3に依頼する例 ===
# 前提: pip install openai が完了していること

from openai import OpenAI
client = OpenAI()

safety_prompt = """
以下のシステムのASIL分類を行ってください：

システム：電動ブレーキアシスト（EBA）
ハザード：ブレーキ力が意図より50%減少する
暴露シナリオ：高速道路走行中（100km/h）
制御可能性：ドライバーが気づいて対処できる確率=60%
損害の深刻さ：追突事故の可能性あり

ISO 26262の4段階評価（ASIL A〜D または QM）で分類し、
各パラメータの根拠とASILの導出過程を示してください。
"""

# reasoning_effort="medium" でコストと精度をバランスさせる
response = client.chat.completions.create(
    model="o3-mini",
    reasoning_effort="medium",
    messages=[{"role": "user", "content": safety_prompt}]
)
print(response.choices[0].message.content)
# 出力例: ASIL Cと分類。暴露=E4, 深刻さ=S3, 制御可能性=C2の根拠も明示される
```

**よくあるエラーと対処：**

| エラー | 原因 | 解決法 |
|--------|------|--------|
| `AuthenticationError` | APIキー未設定 | `OPENAI_API_KEY` 環境変数をセット |
| `RateLimitError` | レート制限超過 | `time.sleep(20)` を挟むか、バッチサイズを減らす |
| `BadRequestError: system role` | o3はsystemロール非対応 | 最初のuserメッセージで役割を定義する |
| 回答が遅い | reasoning_effort="high"の思考時間 | `low`に変更（コストも1/5に） |

## 注意点・落とし穴

- **コスト管理が必須**：o3フルはGPT-4oの約3倍。バッチ処理では `o3-mini` と `reasoning_effort="low"` を基本とし、複雑な問題だけhighにエスカレーションする戦略が効果的
- **system messageが非対応**：o3はシステムプロンプトが使えない。代わりに最初のuserメッセージで「あなたはFEAの専門家です」と役割を定義する
- **思考トークンが別途課金**：`reasoning_tokens` はレスポンストークンとは別に課金される。複雑な問題では意図せず高額になることがある。事前に上限(`max_completion_tokens`)を設定すること
- **自信ある誤答に注意**：推論が深いほど「確信を持った誤答」も出やすい。出力された数値は必ず独立した手段で検証する習慣を持つこと

## 応用：より高度な使い方

o3の推論能力が最も活きるのは**複数の制約が絡み合う問題**だ。Simulink Agentic Toolkitと組み合わせて「現在のモデル構造から潜在的なDivide-by-Zeroを全て列挙させる」、あるいはPolyspace Copilotと連携して「MISRA違反の安全への影響度を定量評価させる」使い方が2026年後半のトレンドになっている。

Anthropic Claudeと比較した場合、o3は数学・論理推論に強く、Claudeは長文文書の要約・コード生成・自然な文章生成に強い。工学的な多段階推論タスクにはo3-mini、ドキュメント生成やコーディングタスクにはClaudeという使い分けが実務で広まりつつある。

## 学生フォーミュラ・レース車両開発への応用

学生フォーミュラチームが最も時間を取られる作業の一つが**サスペンションジオメトリの設計検証**だ。インスタントセンター・ロールセンター高さ・アンチダイブ率の相互依存を手計算で確認する作業は、経験の浅いメンバーには何時間もかかる。

**シナリオ：フロントサスペンションのアンチダイブ率検証**

アンチダイブ（Anti-dive）とは、ブレーキ時に車体前部が沈み込む動きを幾何学的に抑制する設計パラメータ（フロント制動力×重心高さ÷ホイールベースで計算）。多段階の幾何計算が絡み合うため、ミスが多い。

```python
# === o3-miniでアンチダイブ計算を検証させる ===
from openai import OpenAI
client = OpenAI()

geometry_prompt = """
学生フォーミュラカーのフロントサスペンション設計を検証してください：

【設計値】
- ホイールベース: 1600mm、重心高さ: 280mm（静止時）
- フロント荷重配分: 45%、ブレーキ前後配分: 65:35
- アッパーAアーム サイドビュー角度: 15°（後方上がり）
- ロアAアーム サイドビュー角度: 8°（後方上がり）
- タイヤ接地点から仮想ピボットまでの水平距離: 820mm

【要件】
- アンチダイブ率 > 20%
- ロールセンター高さ: 20〜60mmの範囲

アンチダイブ率を計算し、要件を満たすか確認後、
満たさない場合は修正案を示してください。
"""

response = client.chat.completions.create(
    model="o3-mini",
    reasoning_effort="high",
    messages=[{"role": "user", "content": geometry_prompt}]
)
print(response.choices[0].message.content)
```

**学生チームが今すぐ試せる最初のステップ：**

1. OpenAI APIキーを取得する（platform.openai.com、無料クレジットあり）
