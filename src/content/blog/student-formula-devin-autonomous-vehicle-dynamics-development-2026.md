## 学生フォーミュラにおける課題

学生フォーミュラのセットアップ担当チームが直面する典型的な壁があります。

「サーキット走行前に最適なダウンフォース設定を決めたい。でも自前のラップシミュレーターがなく、エクセルで粗い計算をするしかない。1セットの設定検討に4時間かかるため、試せる案が週3〜4通りが限界だ」

ラップシミュレーターを開発しようとしても、プログラミングの経験者が少ない学生チームでは「Pythonでゼロから書く」のが困難です。経験者に頼んでも本業の卒業研究と並行では2〜3ヶ月かかってしまう。数字で示すと：

- 手動エクセル計算：1設定案あたり4時間、1週間で3〜4案しか検討できない
- 自作シミュレーター開発：ゼロから書くと2〜3ヶ月の工数が必要
- 設計締め切りまでの猶予：シーズン前の3〜4ヶ月で他の開発タスクと競合

## Devinを使った解決アプローチ

Devinは「自律AIソフトウェアエンジニア（Autonomous AI Software Engineer）」です。「何を作りたいか」を自然言語で指示すると、ターミナル・コードエディタ・ブラウザを内包した隔離サンドボックス環境でDevinが自律的に設計・実装・テスト・デバッグを完遂します。

なぜこれがラップシミュレーター開発に有効かというと、ラップシミュレーターは「定型化された工学問題」だからです。

ラップシミュレーターの計算構造——質点モデル（Point Mass Model：車を質点として力のつりあいで運動を解析する手法）による車両運動方程式、タイヤのGG-Diagram制約（横Gと縦Gのトレードオフを示す楕円モデル）、コーナー進入・定常旋回・脱出フェーズの解析——はレース工学の教科書に確立されており、Devinはこのパターンをすでに知っています。

つまり「どんな車両パラメータを入力にして何を出力してほしいか」を明確に書くだけで、Devinが数日でコードを完成させてくれます。

## 実装：ステップバイステップ

### 前提条件

- Devinアカウント（[devin.ai](https://devin.ai) から申し込み、月額プラン）
- GitHubアカウント（Devinがコードをリポジトリにpushするため必須）
- 自チームの車両スペックデータ（質量・エンジン出力・タイヤ特性など）

### ステップ1：GitHubリポジトリを準備する

```bash
# GitHubでリポジトリを作成後、URLをメモしておく
# 例: https://github.com/your-team/formula-lap-sim
# （Devinがこのリポジトリにアクセスしてコードをpushします）
```

Devinのダッシュボードで「Connect GitHub」からリポジトリ連携を設定します（5分でできます）。

### ステップ2：Devinへのタスク定義を書く

Devinのチャット画面に以下の形式でタスクを入力します（日本語でOK）。

```
以下の仕様でPythonベースのラップシミュレーターを実装してください。

【目的】
学生フォーミュラ車両の任意のサーキットにおけるラップタイムを予測する。
セットアップ変更（ダウンフォース設定、ギア比など）による性能変化を比較する。

【入力パラメータ（setup.json で指定）】
- 車両質量: m = 280 kg（ドライバー込み）
- 最大エンジン出力: P_max = 65 kW
- 最大横G: a_lat_max = 1.8 G（タイヤグリップ限界）
- 最大縦G（加速）: a_lon_accel = 1.2 G
- 最大縦G（制動）: a_lon_brake = 1.8 G
- ダウンフォース係数: C_L（設定変更可能な変数）
- ドラッグ係数: C_D（設定変更可能な変数）
- サーキットデータ: track.csv（"distance_m" と "corner_radius_m" の列）

【出力】
- 予測ラップタイム（秒）
- セクターごとの速度プロファイル（グラフPNG）
- セットアップA vs B の比較テキストレポート

【技術要件】
- Python 3.10以上、依存ライブラリはpip installで揃えること
- 実行コマンド: python lap_sim.py --track track.csv --setup setup.json
- 日本語コメント付き

リポジトリ: https://github.com/your-team/formula-lap-sim
```

### ステップ3：Devinが自律実行する（待つだけ）

Devinはタスクを受け取ると自律的に作業を開始します。学生は授業中や他の作業をしている間に、Devinがバックグラウンドで以下を実行します。

```
# Devinが自動で行うこと（人間の操作不要）

[1/6] 要件分析と設計
   → 質点法（Point Mass Model）で実装方針を決定
   → コーナー：最大横G制約、加速：エンジン出力制約 を確認

[2/6] コア実装
   → lap_sim.py メインファイル作成
   → vehicle_model.py 車両パラメータクラス
   → track_parser.py CSVパーサー

[3/6] 自動テスト
   → 既知のサーキット例でベンチマーク実行
   → 計算誤差 < 2% であることを確認

[4/6] バグ修正
   → ゼロ除算エラーを検出・修正
   → コーナー接続部のスムージング処理を追加

[5/6] ドキュメント生成
   → README.md（日本語）を自動作成
   → setup.json の設定例を追加

[6/6] PR作成
   → feature/lap-simulator ブランチにpush
   → GitHub PRを自動作成（レビュー待ち状態で通知）
```

### ステップ4：生成されたコードを使う

Devinが作成したPRをマージ後、実際に使います。

```bash
# 依存ライブラリのインストール（Devinが requirements.txt も作ってくれる）
pip install -r requirements.txt

# セットアップ比較の実行例
python lap_sim.py --track track.csv --setup setup_high_df.json
python lap_sim.py --track track.csv --setup setup_low_df.json
```

このコードを実行すると以下が出力されます：

```

  予測ラップタイム: 74.3 秒
  最高速度: 103.2 km/h  コーナー平均: 58.4 km/h

[Low DF]   C_L=1.6, C_D=1.1
  予測ラップタイム: 73.8 秒  ← 0.5秒速い
  最高速度: 108.7 km/h (+5.5)  コーナー平均: 55.1 km/h (-3.3)

推奨: Low DFセットアップが今回のコースに有利
     （ストレート比率が高いコースのためドラッグ低減が効く）
speed_profile.png を出力しました。
```

## よくあるエラーと対処

| エラー | 原因 | 解決法 |
|--------|------|--------|
| PRがいつまでも作成されない | タスク定義が曖昧すぎる | 「入力・出力・技術要件」を明示してタスクを再送信 |
| 生成コードが動かない | 依存ライブラリのバージョン不整合 | エラーメッセージをDevinのチャットに貼り付けてフォローアップ |
| ラップタイムの計算結果が明らかにおかしい | 車両パラメータの単位ミス（kgとN混在など） | setup.jsonの単位をDevinに確認させる |
| サーキットCSVが読めない | 列名が仕様と異なる | track.csvの先頭5行をDevinのチャットに貼り付ける |
| Devinが止まっている | 長時間タスクでセッションタイムアウト | Devinダッシュボードで「Resume」をクリック |

## 学生フォーミュラ・レース車両開発への応用

### シナリオ：テレメトリ自動解析パイプラインの構築

毎回のテスト走行後に `aim_session_*.csv` ファイルが溜まっていくが、セッションをまたいだトレンド分析は手作業でやっている——そんなチームにDevinは特に有効です。

**具体的なシナリオ：**
チームがAiMデータロガーで取得した走行CSV（速度・スロットル開度・G値・ラップタイム）を蓄積している。シーズンを通じた改善トレンドを可視化したいが、複数CSVをまたいだ分析コードを書ける人材がいない。

**Devinへのタスク定義例（そのままコピーして使えます）：**

```
AiM形式のCSVログを複数読み込んで、以下を自動分析・可視化するPythonスクリプトを作成してください。
・セッション間のベストラップタイム推移グラフ
・ブレーキポイントの前後比較（セクター別）
・スロットル開度の走行距離プロファイル比較
出力: HTMLダッシュボード（plotly使用）
CSVフォーマット: Time[s], Speed[km/h], ThrottlePos[%], BrakePress[bar], LapNum
```

**背景理論（学部生向け）：**

テレメトリ解析パイプラインは「データ取り込み → 前処理（欠損値補完・ノイズ除去）→ ラップ区切り検出 → 指標計算 → 可視化」というステップで構成されます。このパターンの設計にはデータエンジニアリングの経験が必要ですが、Devinはこの「定型構造」を学習しています。

**Before / After 数値：**

| 項目 | 手作業 | Devin自律生成後 |
|------|--------|----------------|
| セッション比較グラフ作成 | 3時間/セッション | 10秒（自動実行） |
| 解析できるセッション数/日 | 2〜3件 | 制限なし |
| 担当者の技術要件 | Python中級以上 | タスク記述のみ |

**今すぐ試せる最初のステップ：**

AiMのCSVを1ファイルだけDevinのチャットに貼り付けて、「このデータのフォーマットを解析して、ベストラップを抽出するPythonスクリプトを書いてGitHubにpushして」と指示するだけで、最初の自律実行が始まります。
